75% of the gamer community

asm.js

  • Why not a bytecode spec like, say, LLVM?
    • LLVM is huge
    • LLVM's bytecode VM ~> 20 Mo!
    • Not designed with security in mind
    • JS being higher level: implementation is not constrained by the bytecode spec -> more freedom for optimization
  • What does byte code give us anyway?
    • Ex: ion monkey: ion monkey takes a internal bytecode, deduces the control flow graph from there, and generates the compied code. We could /may in the future move from an AST -> CFG (as v8 does), and that would be possible only because we are not constrained by a bytecode spec. asm.js'd code can call efficiently into JS and vis-vera, boundaries between languages/runtimes are usually very expensive (btw we rewrite certain parts of the dom apis, or js features, in JS (Array.map) which give us better perfs). (on dart: fitting two languages in one VM is HARD and restrict your ability to optim both languages to the max. Better to optimize one language).

"Compiling" to JS

  • Emscripten: LLVM IR → JS
  • Works great with C++
  • Generates very efficient code (asm.js)
  • Performance ~1.5x native performance
  • Very dense representation
  • There's also Mandreel and Duetto
    • OTOY's ORBIX.js video decoder is hand-written code that is almost asm.js

A distribution platform

The web is a standard platform

* You are not completely dependent on one company creates the platform * Browser implementors are working together, differenciation = quality of implems rather than who provides a better API. everyon wants a consensus.

The web is an open platform

Not restricted by the app store model You are not completely dependent on one company that controls distribution * The standardization process is open, if you the standard is not good enough, come help us make it better! * WebAudio was greatly influenced by the needs of the gaming industry

The web is a trusted platform

I don't install a game on my computer unless I trust the source
